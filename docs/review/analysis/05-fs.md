## 1. Purpose and Role

The filesystem stack has three layers:

1. **GPT/MBR** (`gpt.c`) — reads the partition table from LBA 1, validates CRC32, populates an in-memory `struct partition[]`; falls back to MBR if the GPT signature is absent or its CRC fails.
2. **Ext4 driver** (`ext4.c`) — mounts a single ext4 partition (hardcoded index 2), reads the superblock and first group descriptor, resolves paths by walking directory entries, and provides `ext4_read_inode` / `ext4_write_file` / `ext4_list_dir` / `ext4_find_inode`.
3. **VFS shim** (`vfs.c`) — a single function, `vfs_resolve_path`, that canonicalises a relative or absolute path against `current_process->cwd`. There is no vnode table, no mount table, and no type abstraction; syscalls call `ext4_*` directly.

## 2. Data Flow

```
virtio_blk_read
      │
      ▼
gpt_init (LBA 1 → gpt_header, CRC32)
      │  fallback → mbr_init (LBA 0)
      ▼
ext4_init (partition 2 → superblock → GDT block 0)
      │
  ext4_find_inode (path → directory walk → inode number)
      │
  ext4_read_inode (direct / single-indirect / double-indirect)
  ext4_write_file (direct blocks only, 12 max)
  ext4_list_dir   (directory entry walk)
      ▲
syscall_dispatch (cases 251–256)
      │ vfs_resolve_path
      ▼
userland process
```

## 3. Explicit Prior-Concern Verdicts

### GPT CRC32 — REFUTED (correct as implemented)

The kernel CRC-32 implementation (`kernel/lib/crc32.c:8–18`) uses the reflected IEEE-802.3 polynomial 0xEDB88320, initialises to 0xFFFFFFFF, and inverts the final result — the standard PKZIP/ISO-3309 variant. The `mkdisk.c` tool uses an identical implementation (`tools/mkdisk.c:195–201`) and sets `header_crc32 = crc32(&h, 92)` after zeroing the CRC field (`tools/mkdisk.c:456`). `gpt_init` reproduces the same zeroing-before-compute sequence (`gpt.c:84–87`). The polynomial, init value, and byte order are consistent between disk creation and verification. **This concern is unfounded; the CRC32 logic is correct.**

### Ext4 Group-Descriptor "Corruption" — PARTIALLY CONFIRMED (latent on test image, real on multi-group)

`struct ext4_group_desc` (`ext4.h:80–89`) is declared as:

```
bg_block_bitmap_lo  (4) +  bg_inode_bitmap_lo (4) +  bg_inode_table_lo (4)
bg_free_blocks_count_lo (2) + bg_free_inodes_count_lo (2) +
bg_used_dirs_count_lo (2) + bg_flags (2) + padding[14]
= 4+4+4+2+2+2+2+14 = 34 bytes
```

The comment "Pad to 32 bytes" is wrong: the named fields sum to 20 bytes and `padding[14]` brings the total to **34 bytes, not 32**. The real ext4 on-disk GDT entry for a non-64-bit filesystem is exactly 32 bytes; a 64-bit filesystem uses 64 bytes. Two consequences:

1. **Read path**: `memcpy(&bg, k_buf, sizeof(bg))` (`ext4.c:170`) copies 34 bytes from the 512-byte sector buffer, ingesting 2 bytes past the 32-byte entry boundary into `padding`. This is harmless.
2. **Write path**: `memcpy(bg_buf, &bg, sizeof(bg))` (`ext4.c:98`) copies 34 bytes into position 0 of the 512-byte GDT sector buffer, then writes the whole sector back. The 2 extra bytes overwrite bytes 32–33 of the GDT sector, which are the first 2 bytes of GDT entry 1 (`bg_block_bitmap_lo` low bytes). **On a single-group filesystem (the current 1024-inode test image), there is no GDT entry 1, so no corruption occurs.** On a multi-group image, every write operation would corrupt the first 2 bytes of GDT entry 1. **The concern is real but latent on the current test image.**

Additionally, if `s_feature_incompat` has `INCOMPAT_64BIT` set, GDT entries are 64 bytes and the `_lo` fields have `_hi` counterparts — the driver silently reads only 34 bytes, losing the high 32 bits of all block/inode addresses. `s_feature_incompat` is modeled in the superblock (`ext4.h:121`) but never read or enforced.

### Volume Label Empty — NOT A BUG

`s_volume_name` in `struct ext4_superblock` is at struct offset 120 = 0x78 (verified by counting all preceding fields). The real ext4 specification places `s_volume_name` at offset 0x78 from the start of the superblock. The offset is correct. The logged `Vol=` (empty string) faithfully reflects an unlabeled volume in the test image.

### `ext4_write_file` — IMPLEMENTED; PERSISTS TO DISK; UNSAFE

`ext4_write_file` (`ext4.c:447–544`) is a real, data-persisting implementation: it allocates blocks via `ext4_alloc_block` (`ext4.c:34–125`), issues `virtio_blk_write` calls (`ext4_bwrite`, `ext4.c:27`), and calls `ext4_update_inode` to persist the updated inode (`ext4.c:538`). The file header ("Simplified Read-Only Ext4 Driver", `ext4.c:3`) and any system-level claim of "read-only for userland" are **false**. Write is functional and exercises the actual block device.

## 6. Detailed Entries

### EXT4-01 — Extent-tree format undetected: silent data corruption on real ext4 images `[static]`

Modern `mkfs.ext4` enables the extent feature by default (`INCOMPAT_EXTENTS`, bit 0x40 of `s_feature_incompat`). When extents are enabled, each inode's `i_block[0..3]` holds a 60-byte extent-tree header (magic 0xF30A, not a block pointer) rather than the classic direct/indirect block map. `ext4_read_inode` (`ext4.c:278–316`) unconditionally interprets `i_block[block_idx]` as a physical block number; `i_flags` is modeled in the struct (`ext4.h:103`) but never read anywhere in `ext4.c` (confirmed: zero matches for `i_flags` in `ext4.c`). Consequence: on any image built with standard `mkfs.ext4`, every data read returns garbage or crashes the virtio layer with an out-of-range sector address.

The bundled `mkdisk.c` tool (`tools/mkdisk.c`) builds block-mapped inodes manually (zero matches for "extent" in `mkdisk.c`) and the test runtime confirms successful mounts, so the test image uses the non-extent, indirect-block format. The defect is therefore **latent on the current test image but fatal on any real-world ext4 filesystem** — it directly blocks the goal of mounting a genuine ext4 partition.

*Fix direction:* Before accessing `i_block[]`, check `inode.i_flags & 0x80000` (`EXT4_EXTENTS_FL`). If set, parse the extent tree (header at `i_block[0]`, magic check `0xF30A`, then index/leaf nodes). This is the primary read-path gap to resolve before mounting images not produced by `mkdisk.c`.

### EXT4-02 — No access control: any process can overwrite any file `[static]`

Syscall 251 (`FILE_WRITE`, `syscall_dispatch.c:176–199`) resolves the path with `vfs_resolve_path` and passes it directly to `ext4_write_file`. There is no check of `current_process->pid`, no file permission model, no notion of file ownership, and no read-only enforcement. The system comment "read-only for userland" is contradicted by the live implementation: any userland process with knowledge of a filename can overwrite it, including `/init`, the shell binary, or the kernel ELF image if stored on the ext4 partition. This directly contradicts the stated seL4-style isolation goal.

*Fix direction (near term):* A per-process UID/GID against inode `i_uid`/`i_gid` check, or a simple "no write from userland" gate, before dispatching to `ext4_write_file`. Long term: capability-based access in the seL4 service model (see §7).

---

### EXT4-04 — Group-descriptor struct is 34 bytes, not 32; multi-group write corruption `[static]`

*Arithmetic:* `struct ext4_group_desc` (`ext4.h:80–89`): 4+4+4+2+2+2+2 = 20 named bytes, plus `padding[14]` = **34 bytes total**. The comment "Pad to 32 bytes" is incorrect by 2 bytes. The on-disk 32-byte GDT entry format is:

| Offset | Size | Field |
|--------|------|-------|
| 0x00 | 4 | bg_block_bitmap_lo |
| 0x04 | 4 | bg_inode_bitmap_lo |
| 0x08 | 4 | bg_inode_table_lo |
| 0x0C | 2 | bg_free_blocks_count_lo |
| 0x0E | 2 | bg_free_inodes_count_lo |
| 0x10 | 2 | bg_used_dirs_count_lo |
| 0x12 | 2 | bg_flags |
| 0x14–0x1F | 12 | bg_exclude_bitmap_lo, checksums, bg_itable_unused_lo, bg_checksum |

The driver's `padding[14]` at offset 20 covers bytes 0x14–0x21, 2 bytes past the 32-byte boundary. On the read path (`ext4.c:170`), this ingests 2 "extra" bytes harmlessly into padding. On the write path (`ext4.c:98`): `memcpy(bg_buf, &bg, sizeof(bg))` copies 34 bytes starting at position 0 of the 512-byte `bg_buf`, then `ext4_bwrite(part_start_lba + 8, 1, bg_buf)` writes the entire 512-byte sector. Bytes 32–33 of the sector (the first 2 bytes of GDT entry 1) are overwritten with 2 bytes from `bg.padding`. On a single-group filesystem (the test image, 1024 inodes), GDT entry 1 does not exist, so no on-disk data is corrupted. On a multi-group image, every `ext4_alloc_block` call silently corrupts `bg_block_bitmap_lo[0:1]` of block group 1.

*Fix:* Set `padding[12]` (not `[14]`) to bring the struct to exactly 32 bytes, matching the on-disk entry size.

### EXT4-05 — Write ceiling is 48 KB; read/write asymmetry is undocumented `[static]`

`ext4_write_file` (`ext4.c:485–489`) rejects writes to `block_idx >= 12` with an error message "Indirect block write not supported yet". The ceiling is 12 direct blocks × 4096 bytes = **49,152 bytes (48 KB)**. `ext4_read_inode` supports direct (block_idx 0–11), single-indirect (12–1035), and double-indirect (1036–1,049,611), covering a theoretical maximum of:

- Direct: 12 × 4096 = 49,152 bytes
- + Single-indirect: 1024 × 4096 = 4,194,304 bytes
- + Double-indirect: 1024 × 1024 × 4096 = 4,294,967,296 bytes
- Total ≈ **4,100 MB** (bounded by the 32-bit `i_size_lo` field at 4 GB)

Triple-indirect (`i_block[14]`) is not implemented; any access beyond block index 1,049,611 returns `-1` (`ext4.c:299`). The function comment at line 234 ("Supports Direct Blocks (0-11) and Single Indirect Blocks (12)") is stale — the implementation reaches double-indirect (`ext4.c:291–316`).

---

### EXT4-07 — Unbounded `kmalloc(size)` from user-controlled argument `[static]`

`syscall_dispatch.c:186`: `uint8_t *k_buf = kmalloc(size)` where `size = (size_t)arg2` is taken directly from the userland register frame. No upper bound is applied. A malicious or buggy process can pass `size = 0xFFFF_FFFF`, causing `kmalloc` to return `NULL` (safe) or, on a kernel heap implementation without an upper-bound guard (see MM-KM-01), to exhaust the 32 MB heap pool permanently. The same pattern appears for FILE_READ (`syscall_dispatch.c:217`) and LIST_DIR (`syscall_dispatch.c:246`). A simple `if (size > MAX_FS_IO_SIZE) return -1` guard before `kmalloc` closes this.

---

### EXT4-08 — Unsigned overflow in read-size clamp `[static]`

`ext4_read_inode` (`ext4.c:249`): `if (offset + size > file_size) size = file_size - offset;`. If `offset` and `size` are both close to `UINT32_MAX`, `offset + size` wraps to a small value, the condition is false, and the function proceeds to read with the oversized `size`. The result is a `virtio_blk_read` into a kernel buffer at an address and size that the caller did not intend. *Fix:* check `size > file_size - offset` (avoiding the add) or use `uint64_t` for the intermediate comparison.

---

### GPT-01 — Partition-entry CRC mismatch is logged but not fatal `[static]`

`gpt.c:139–142`: when `calc_entries_crc != entries_crc`, the driver logs the mismatch but falls through to the partition-parsing loop (`gpt.c:145–178`). Contrast with the header CRC mismatch at `gpt.c:89–101`, which correctly aborts and falls back to MBR. Parsing entries whose integrity cannot be confirmed can populate `partitions[]` with corrupt start/end LBAs, causing subsequent reads to access arbitrary sectors on the block device.

*Fix:* add `return;` after the `pr_err` at `gpt.c:142` (or fall through to `mbr_init` as the header-CRC path does).

---

### VFS-01 — VFS is a stub path normaliser; no mountable abstraction exists `[static]`

The entire "VFS" layer is a single 59-line file (`vfs.c`) containing one function, `vfs_resolve_path`. There is no:
- vnode or inode abstraction visible to the kernel (callers use raw ext4 inodes).
- Mount table or mount-point concept.
- Filesystem-type dispatch (`struct file_operations` or equivalent).
- File-descriptor table above the syscall layer.
- Any hook for mounting a second filesystem type.

All FS syscalls in `syscall_dispatch.c` (cases 251–256) call `ext4_*` directly, with `extern` declarations at lines 41–44. The Plan 9 "everything is a file" namespace and the seL4 userland-service goal both require a real VFS layer as a prerequisite. The distance between the current state and that goal is large (see §7).

## 7. Refactor Direction (toward the declared goals)

| Goal | FS implication |
|------|---------------|
| **Mount real ext4 images** | EXT4-01 (extent support) is the single blocking gap. |
| **seL4-style isolation** | VFS-01 is the structural prerequisite: a vnode/file-ops table decouples the capability-granting server from the kernel; EXT4-02 (access control) must precede any real isolation. |
| **Plan 9 "everything is a file"** | Requires a mount table and a `struct file_operations`-style dispatch so that `/dev`, `/proc`, `/net` endpoints can be mounted as filesystem backends alongside ext4. VFS-01 is the root task. |
| **Write correctness on multi-group images** | EXT4-04 (GDT struct size), EXT4-05 (indirect write), EXT4-06 (feature flags), and missing journal bypass. |
| **Security** | EXT4-02, EXT4-07, and VFS-01 (capability enforcement at the VFS layer). |
| **Performance** | EXT4-11 (indirect block re-reads), EXT4-15 (buffer-cache bypass); hooking into `buffer.c` for block caching would address both. |

**Suggested sequencing:**

1. **EXT4-01** — add `EXT4_EXTENTS_FL` check to `ext4_read_inode` before any other ext4 work, so the driver correctly signals unsupported format rather than silently corrupting reads.
2. **EXT4-04** — fix group-descriptor struct size (trivial 1-line fix, avoids future multi-group corruption).
3. **GPT-01** — add abort on partition-entry CRC mismatch.
4. **EXT4-02 / EXT4-07** — harden the syscall layer (size bound, per-process write gate).
5. **VFS-01** — introduce `struct vnode`, `struct mount`, and a `struct file_ops` dispatch table; redirect syscalls through the table. This unblocks Plan 9 namespace and seL4 service work.
6. **EXT4-05 / EXT4-06** — indirect write support and feature-flag enforcement once the VFS layer is in place.
7. **EXT4-01 (extent tree)** — full extent-tree parser to support real-world ext4 images.
8. **EXT4-15** — integrate with `buffer.c` for block caching.

**Long-term (seL4 / Plan 9):** The VFS layer should become a userland capability server — a process that holds the block device capability and exposes a 9P-like protocol. Kernel VFS code becomes thin stubs that IPC into the server. The `vfs_resolve_path` function is already the seed of a namespace resolver; a `vnode` table with `open/read/write/readdir/close` ops is the next concrete artefact.
