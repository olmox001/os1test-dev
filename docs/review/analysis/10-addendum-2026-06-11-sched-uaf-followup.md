## 1. Verdict on the reviewed fixes

**SCHED-UAF-01 (generic + aarch64) and ARCH-AMD64-APPGD-01 are correct.** [verified]

- `virt_to_phys` is identity (`kernel/include/kernel/vmm.h:144`), so the
  `(uint64_t)kernel_pgd` casts in both `arch_cpu_switch_context` implementations
  are exactly equivalent to the removed `virt_to_phys()` call.
- Init ordering supports the AP adoption: `init_memory()` (`kernel/main.c:107`,
  runs `vmm_init` + `vmm_dynamic_remap`) precedes `arch_smp_init()`
  (`kernel/main.c:123`); APs enable IRQs only *after* `cpu_init()`
  (`kernel/main.c:240-245`), so no device IRQ can land before CR3 adoption.
- The reaped PGD is freed only in the *next* `schedule()` drain on the owning
  CPU, by which point that CPU has loaded a different (live) PGD — the UAF
  window is closed.
- aarch64 secondaries are unaffected by the APPGD class: `platform.c:207`
  publishes the *post-remap* TTBR0 to `secondary_ttbr0` at SMP init.
- The feared TLB-flush perf regression is a non-issue: the old generic block
  already broadcast-flushed on every PGD change, and every real switch changes
  the PGD (the `prev == next` case returns early).

## 3. Residual issues — known, NOT fixed here (newly catalogued)

These were *detected* by this review but deliberately not fixed in this pass
(each needs its own scoped change + runtime verification):

- **SCHED-IRQ-01 (W3 BUG/WRONG-DESIGN, new).** `schedule()` has no IRQ-state
  contract: the timer path enters it with IRQs masked, but several syscall
  paths enter it with IRQs **enabled** (`syscall_dispatch.c:239/248`, `:265`,
  `:493/496`; aarch64 `syscall.c:201`). SCHED-UAF-02 (above) closed the
  concrete double-free, but the prologue between `schedule()` entry and
  `spin_lock_irqsave` (reading `current_task`, `compositor_get_focus_pid()`)
  is still re-entrant by design-accident. Correctness currently rests on the
  nested invocation restoring `current_task == prev` before the outer frame
  resumes. *Fix direction:* make "IRQs masked" a precondition of `schedule()`
  (mask in the syscall dispatcher before yielding, or `local_irq_save` for the
  whole function) — one mechanism, not per-caller patches. [static]
- **TOOLING-03 (W1 PROCESS).** The amd64 toolchain is unpinned: the script
  installs whatever `x86_64-elf-gcc` Homebrew ships (16.1.0 known-good today).
  A future major bump can silently change codegen for `-mno-red-zone -mcmodel=kernel`
  paths. *Fix direction:* record the verified version and have `make check`
  warn on drift (aarch64 already has the 7.2.0 pin + atomics probe).
- **MM-VMM (existing TODO, now load-bearing).** `vmm_dynamic_remap` leaks
  `old_pgd` intentionally (`kernel/mm/vmm.c:439-443`, needs IPI + TLB-shootdown
  to free safely). With the scheduler now adopting `kernel_pgd` on every
  kernel-thread switch, that one page is permanently referenced history —
  harmless, but the TODO's "after all secondary CPUs have transitioned"
  condition is *already met* at the end of `arch_smp_init()`; a simple
  free-at-that-point is now viable.

## 4. Residual issues — undetected by this diff-scoped review (where to look next)

A diff review cannot clear the whole tree. The highest-value known-open areas,
consolidated from [`REVIEW.md`](../REVIEW.md) §8-§9, the in-tree `NOTE()` tag
catalog (~80 distinct IDs), and the Phase-A plan
([`fault-handling-backtrace-review.txt`](../../fault-handling-backtrace-review.txt)):

1. **Fault handling is still halt-only on amd64** (EXC-AMD64-02, W4): every
   exception → dump + `arch_cpu_halt()`; no user/kernel discrimination, no
   process termination, no IST for #DF/#PF (`gdt.c` `ist[7]` all-zero,
   `idt.c:143/149` gates with `ist=0`). The window-close crash family stays
   one bug away from a reboot until Phase A lands.
2. **LAPIC-MMIO `get_cpu_id` inside fault paths** (Phase-A landmine, W3):
   `arch_impl_get_cpu_id` dereferences `0xFEE00020`; on a compromised address
   space the #PF handler's own `pr_err` path can fault recursively. Must read
   from `%gs` per-CPU data instead.
3. **CPU-AMD64-01 (W3):** SSE enabled but no FXSAVE/XSAVE in the preemptive
   ISR switch path — silent user FP/XMM state corruption under preemption.
4. **Scheduler↔compositor coupling** (SCHED-01/SCHED-03/GFX-COMP-03, W3
   WRONG-DESIGN): `compositor_get_focus_pid()` on every `schedule()`;
   `process_terminate` from mouse-IRQ context.
5. **No W^X anywhere** (MM-VMM-01/AMMU-01, W3 SECURITY) and the PA==VA
   single-address-space invariant (higher-half rework) — the two structural
   security items.
6. **aarch64 silent-hang stubs** (W2): all FIQ and AArch32-EL0 vectors are
   `b .` infinite loops (`exception.S:283-328`) — a wedged core with no
   diagnostic if ever taken.
7. **IPS=32-bit on aarch64** (`arch_vmm_init_hw`, W2): TCR caps PA at 4GB —
   fine for QEMU virt, undersized for real hardware.

Items 1-2 are exactly the Phase A ("fault/trace foundation") scope already
planned in the backtrace review doc; 3-7 are tracked in `REVIEW.md` / GitHub
issues and should be scheduled after Phase A.
