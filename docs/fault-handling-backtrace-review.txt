This is a crucial landmine for the plan: arch_impl_get_cpu_id() on amd64 dereferences LAPIC MMIO at 0xFEE00020. If a fault occurs while the CPU is on a freed/compromised PGD, the LAPIC mapping may be gone, so calling arch_get_cpu_id() inside the fault handler (which amd64_page_fault_handler does at idt.c:202) can itself fault — compounding the recursion. The fault handler must read the CPU id from %gs: (cpu_info) instead, which is set via MSR and doesn't require a page-table walk.

PART 4 — RISKS / LANDMINES
IST stack sizing & count (amd64, Step 3). Too-small IST → the fault handler + backtrace walk overflows the IST itself → #DF on the #DF stack → triple fault. Use ≥16KB, and a separate IST index for #DF vs #PF. Per-CPU × MAX_CPUS(64) inflates .bss — consider sizing IST arrays to nr_cpus actually used, or a smaller MAX for IST.
swapgs on nested faults (amd64). If a fault occurs in kernel mode between swapgs and the GS-based logic (or in the syscall entry window), %gs may be the user base. The Step-5 %gs:cpu_id read and the in_fault guard assume kernel GS. An IST entry does not auto-swapgs. Document that fault handlers must not assume GS is kernel unless swapgs ran; prefer reading cpu_id from a source that works regardless, or gate on CS. NMI is the classic case — if NMI is given an IST, its swapgs handling needs the "fault from kernel that already had kernel GS" paranoia (Linux's paranoid-entry pattern). Keep NMI out of scope unless needed.
FXSAVE/XMM in the fault path (CPU-AMD64-01, interacts with Step 9). amd64 common_isr_entry saves no XMM. When a user fault now calls schedule() → context switch (Step 9), XMM state of the preempted/terminated task isn't saved — for a terminated task that's harmless, but if the helper ever returns to a live task via schedule on a preempt path, XMM could corrupt. Confine Step 9's schedule() return to the terminate path; do not introduce new preemptive switches without addressing FXSAVE.
Frame-pointer reliability across asm boundaries (Step 11). isr_stubs.S, exception.S, context.S, syscall.S don't build standard frames; ctx_switch (context.S:12) saves rbp but doesn't chain it as a frame. Seed the walk from regs (rip+rbp), not from the handler's own frame, and stop cleanly when fp leaves the kernel-stack range. -O2 may elide frames for small/leaf fns despite -fno-omit-frame-pointer (the flag forces the prologue but tail-calls still merge frames) — accept occasional skipped frames.
Symbol-table size & placement (Step 12). Full .symtab is large; filter to text symbols and compress. The blob must be ALLOC + in .rodata or aarch64's objcopy -O binary kernel.bin drops it (the running aarch64 image is raw binary). Two-pass link can break the build mid-development — gate it so the default single-pass build always boots.
Backtrace dereferences faulting (Steps 11/12). The walk reads stack memory and the symbol blob; a bad fp must hit the Step-7 in_fault guard (clean halt), never recurse. Bound-check every fp before dereference.
aarch64 abort-stack switch (Step 4). Switching SP in the vector before saving the frame means the original faulting SP must be captured into the frame for an accurate dump; and the abort stack is itself re-entrant — rely on the in_fault guard to stop a second-level abort.
Generic schedule() PGD change (Step 2) affects both arches at once. A mistake here breaks both boots simultaneously. Land Step 1 (aarch64 HAL) first and verify, then make the generic change with both arches tested in the same commit.
uart_lock-free emergency path (Step 6) can interleave with a concurrent normal printk on another CPU (garbled output). That's an acceptable trade for never deadlocking in a fault; the panic IPI (panic() line 222) should still quiesce other CPUs first when the kernel is healthy enough to send it.
Maintainer constraint: do not touch kernel/arch/amd64/platform/platform.c. None of the steps above require it. Generic changes (Steps 2, 6, 7, 8, 11, 12, 15) are explicitly flagged and are the only non-HAL edits; all are in shared subsystems where the logic genuinely belongs.
